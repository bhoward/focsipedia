Finite-State Automata

(Content adapted from Critchlow & Eck)

We have seen how regular expressions can be used to generate languages mechanically. How might languages be recognized mechanically? The question is of interest because if we can mechanically recognize languages like L={all legal C++ programs that will not go into infinite loops on any input}L= \{\textrm{all legal C++ programs that will not go into infinite loops on any input}\}L={all legal C++ programs that will not go into infinite loops on any input}, then it would be possible to write über-compilers that can do semantic error-checking like testing for infinite loops, in addition to the syntactic error-checking they currently do.

What formalism might we use to model what it means to recognize a language "mechanically"? We look for inspiration to a language-recognizer with which we are all familiar, and which we've already in fact mentioned: a compiler. Consider how a C++ compiler might handle recognizing a legal if statement. Having seen the word if, the compiler will be in a state or phase of its execution where it expects to see a (; in this state, any other (non-blank) character will put the compiler in a "failure" state. If the compiler does in fact see a ( next, it will then be in an "expecting a boolean condition" state; if it sees a sequence of symbols that make up a legal boolean condition, it will then be in an "expecting a )" state; and then "expecting a { or a legal statement"; and so on. Thus one can think of the compiler as being in a series of states; on seeing a new input symbol, it moves on to a new state; and this sequence of transitions eventually leads to either a "failure" state (if the if statement is not syntactically correct) or a "success" state (if the if statement is legal). We isolate these three concepts—states, input-inspired transitions from state to state, and "accepting" vs "non-accepting" states—as the key features of a mechanical language-recognizer, and capture them in a model called a finite-state automaton. (Whether this is a successful distillation of the essence of mechanical language recognition remains to be seen; the question will be taken up later in this chapter.)

A finite-state automaton (FSA), then, is a machine which takes, as input, a finite string of symbols from some alphabet Σ\SigmaΣ. There is a finite set of states in which the machine can find itself. The state it is in before consuming any input is called the start state. Some of the states are accepting or final. If the machine ends in such a state after completely consuming an input string, the string is said to be accepted by the machine. The actual functioning of the machine is described by something called a transition function, which specifies what happens if the machine is in a particular state and looking at a particular input symbol. ("What happens" means "in which state does the machine end up".)

FSAs actually come in two flavors depending on what properties you require of the transition function. We will look first at a class of FSAs called deterministic finite-state automata (DFAs). In these machines, the current state of the machine and the current input symbol together determine exactly which state the machine ends up in: for every ⟨current state, current input symbol⟩\langle\textrm{current state, current input symbol}\rangle⟨current state, current input symbol⟩ pair, there is exactly one possible next state for the machine.

Formally, a deterministic finite-state automaton MMM is specified by 5 components: M=(Q,Σ,q0,δ,F)M=(Q, \Sigma, q_0, \delta, F)M=(Q,Σ,q0​,δ,F) where

  • QQQ is a finite set of states;
    • Σ\SigmaΣ is an alphabet called the input alphabet;
    • q0∈Qq_0 \in Qq0​∈Q is a state which is designated as the start state;
    • FFF is a subset of QQQ; the states in FFF are states designated as final or accepting states;
    • δ\deltaδ is a transition function that takes ⟨state, input symbol⟩\langle\textrm{state, input symbol}\rangle⟨state, input symbol⟩ pairs and maps each one to a state: δ:Q×Σ→Q\delta : Q \times \Sigma \rightarrow Qδ:Q×Σ→Q. To say δ(q,a)=q′\delta(q,a) = q'δ(q,a)=q′ means that if the machine is in state qqq and the input symbol aaa is consumed, then the machine will move into state q′q'q′. The function δ\deltaδ must be a total function, meaning that δ(q,a)\delta(q,a)δ(q,a) must be defined for every state qqq and every input symbol aaa. (Recall also that, according to the definition of a function, there can be only one output for any particular input. This means that for any given qqq and aaa, δ(q,a)\delta(q,a)δ(q,a) can have only one value. This is what makes the finite-state automaton deterministic: given the current state and input symbol, there is only one possible move the machine can make.)

The transition function δ\deltaδ describes only individual steps of the machine as individual input symbols are consumed. However, we will often want to refer to "the state the automaton will be in if it starts in state qqq and consumes input string www", where www is a string of input symbols rather than a single symbol. Following the usual practice of using ∗^*∗ to designate "0 or more", we define δ∗(q,w)\delta^*(q,w)δ∗(q,w) as a convenient shorthand for "the state that the automaton will be in if it starts in state qqq and consumes the input string www". For any string, it is easy to see, based on δ\deltaδ, what steps the machine will make as those symbols are consumed, and what δ∗(q,w)\delta^*(q,w)δ∗(q,w) will be for any qqq and www. Note that if no input is consumed, a DFA makes no move, and so δ∗(q,ε)=q\delta^*(q, \varepsilon) = qδ∗(q,ε)=q for any state qqq.1

We have divided the states of a DFA into accepting and non-accepting states, with the idea that some strings will be recognized as "legal" by the automaton, and some not. Formally:

(Don't get confused by the notation. Remember, it's just a shorter and neater way of saying "w∈Σ∗w \in \Sigma^*w∈Σ∗ is accepted by MMM if and only if the state that MMM will end up in if it starts in q0q_0q0​ and consumes www is one of the states in FFF.")

The language accepted by MMM, denoted L(M)L(M)L(M), is the set of all strings w∈Σ∗w \in \Sigma^*w∈Σ∗ that are accepted by MMM: L(M)={w∈Σ∗ ∣ δ∗(q0,w)∈F}L(M) = \{ w \in\Sigma^* \ | \ \delta^*(q_0, w) \in F\}L(M)={w∈Σ∗ ∣ δ∗(q0​,w)∈F}.

Note that we sometimes use a slightly different phrasing and say that a language LLL is accepted by some machine MMM. We don't mean by this that LLL and maybe some other strings are accepted by MMM; we mean L=L(M)L = L(M)L=L(M), i.e., LLL is exactly the set of strings accepted by MMM.

As well as recognizing what language is accepted by a given DFA, we often want to do the reverse and come up with a DFA that accepts a given language. Building DFAs for specified languages is an art, not a science. There is no algorithm that you can apply to produce a DFA from an English-language description of the set of strings the DFA should accept. On the other hand, it is not generally successful, either, to simply write down a half-dozen strings that are in the language and design a DFA to accept those strings—invariably there are strings that are in the language that aren't accepted, and other strings that aren't in the language that are accepted. So how do you go about building DFAs that accept all and only the strings they're supposed to accept? The best advice I can give is to think about relevant characteristics that determine whether a string is in the language or not, and to think about what the possible values or "states" of those characteristics are; then build a machine that has a state corresponding to each possible combination of values of relevant characteristics, and determine how the consumption of inputs affects those values. I'll illustrate what I mean with a couple of examples.

Example: Find a DFA with input alphabet Σ={a,b}\Sigma = \{a,b\}Σ={a,b} that accepts the language L={w∈Σ∗ ∣ na(w) and nb(w) are both even }L= \{w \in\Sigma^* \ | \ n_a(w) \textrm{ and } n_b(w) \textrm{ are both even } \}L={w∈Σ∗ ∣ na​(w) and nb​(w) are both even }.

The characteristics that determine whether or not a string www is in LLL are the parity of na(w)n_a(w)na​(w) and nb(w)n_b(w)nb​(w). There are four possible combinations of "values" for these characteristics: both numbers could be even, both could be odd, the first could be odd and the second even, or the first could be even and the second odd. So we build a machine with four states q1,q2,q3,q4q_1, q_2, q_3, q_4q1​,q2​,q3​,q4​ corresponding to the four cases. We want to set up δ\deltaδ so that the machine will be in state q1q_1q1​ exactly when it has consumed a string with an even number of aaa's and an even number of bbb's, in state q2q_2q2​ exactly when it has consumed a string with an odd number of aaa's and an odd number of bbb's, and so on.

To do this, we first make the state q1q_1q1​ into our start state, because the DFA will be in the start state after consuming the empty string ε\varepsilonε, and ε\varepsilonε has an even number (zero) of both aaa's and bbb's. Now we add transitions by reasoning about how the parity of aaa's and bbb's is changed by additional input. For instance, if the machine is in q1q_1q1​ (meaning an even number of aaa's and an even number of bbb's have been seen) and a further aaa is consumed, then we want the machine to move to state q3q_3q3​, since the machine has now consumed an odd number of aaa's and still an even number of bbb's. So we add the transition δ(q1,a)=q3\delta(q_1, a) = q_3δ(q1​,a)=q3​ to the machine. Similarly, if the machine is in q2q_2q2​ (meaning an odd number of aaa's and an odd number of bbb's have been seen) and a further bbb is consumed, then we want the machine to move to state q3q_3q3​ again, since the machine has still consumed an odd number of aaa's, and now an even number of bbb's. So we add the transition δ(q2,b)=q3\delta(q_2, b) = q_3δ(q2​,b)=q3​ to the machine. Similar reasoning produces a total of eight transitions, one for each state-input pair. Finally, we have to decide which states should be final states. The only state that corresponds to the desired criteria for the language LLL is q1q_1q1​, so we make q1q_1q1​ a final state. Example: Find a DFA with input alphabet Σ={a,b}\Sigma = \{a,b\}Σ={a,b} that accepts the language LLL = {w∈Σ∗ ∣ na(w) is divisible by 3 }\{w\in\Sigma^* \ | \ n_a(w) \textrm{ is divisible by 3 } \}{w∈Σ∗ ∣ na​(w) is divisible by 3 }.

The relevant characteristic here is of course whether or not the number of aaa's in a string is divisible by 3, perhaps suggesting a two-state machine. But in fact, there is more than one way for a number to not be divisible by 3: dividing the number by 3 could produce a remainder of either 1 or 2 (a remainder of 0 corresponds to the number in fact being divisible by 3). So we build a machine with three states q0q_0q0​, q1q_1q1​, q2q_2q2​, and add transitions so that the machine will be in state q0q_0q0​ exactly when the number of aaa's it has consumed is evenly divisible by 3, in state q1q_1q1​ exactly when the number of aaa's it has consumed is equivalent to 1 mod 31 \bmod{3}1mod3, and similarly for q2q_2q2​. State q0q_0q0​ will be the start state, as ε\varepsilonε has 0 aaa's and 0 is divisible by 3. The completed machine is shown below. Notice that because the consumption of a bbb does not affect the only relevant characteristic, bbb's do not cause changes of state.

Example: Find a DFA with input alphabet Σ={a,b}\Sigma = \{a,b\}Σ={a,b} that accepts the language LLL = {w∈Σ∗ ∣w contains three consecutive a’s }\{w\in\Sigma^* \ | w \textrm{ contains three consecutive a's } \}{w∈Σ∗ ∣w contains three consecutive a’s }.

Again, it is not quite so simple as making a two-state machine where the states correspond to "have seen aaaaaaaaa" and "have not seen aaaaaaaaa". Think dynamically: as you move through the input string, how do you arrive at the goal of having seen three consecutive aaa's? You might have seen two consecutive aaa's and still need a third, or you might just have seen one aaa and be looking for two more to come immediately, or you might just have seen a bbb and be right back at the beginning as far as seeing 3 consecutive aaa's goes. So this time there will be four states, with the "last symbol was not an aaa" state being the start state. The complete automaton is shown below.
