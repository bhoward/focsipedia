Computability

(Content adapted from Critchlow & Eck)

At this point, it would be useful to look at increasingly complex Turing machines, which compute increasingly complex functions and languages. Although Turing machines are very simple devices, it turns out that they can perform very sophisticated computations. In fact, any computation that can be carried out by a modern digital computer—even one with an unlimited amount of memory—can be carried out by a Turing machine. Although it is not something that can be proved, it is widely believed that anything that can reasonably be called "computation" can be done by a Turing machine. This claim is known as the Church-Turing Thesis.

We do not have time to look at enough examples to convince you that Turing machines are as powerful as computers, but the proof reduces to the fact that computers are actually fairly simple in their basic operation. Everything that a computer does comes down to copying data from one place to another, making simple comparisons between two pieces of data, and performing some basic arithmetic operations. It's possible for Turing machines to do all these things. In fact, it's possible to build a Turing machine to simulate the step-by-step operation of a given computer. Doing so proves that the Turing machine can do any computation that the computer could do, although it will, of course, work much, much more slowly.

Extended Turing Machines#

We can, however, look briefly at some other models of computation and see how they compare with Turing machines. For example, there are various ways in which we might try to increase the power of a Turing machine. For example, consider a two-tape Turing machine that has two tapes, with a read/write head on each tape. In each step of its computation, a two-tape Turing machine reads the symbols under its read/write heads on both tapes. Based on these symbols and on its current state, it can write a new symbol onto each tape, independently move the read/write head on each tape one cell to the left or right, and change state.

It might seem that with two tapes available, two-tape Turing machines might be able to do computations that are impossible for ordinary one-tape machines. In fact, though, this is not the case. The reason, again, is simulation: Given any two-tape Turing machine, it is possible to build a one-tape Turing machine that simulates the step-by-step computation of the two-tape machine. Let MMM be a two-tape Turing machine. To simulate MMM with a one-tape machine, KKK, we must store the contents of both of MMM's tapes on one tape, and we must keep track of the positions of both of MMM's read/write heads. Let @ and $ be symbols that are not in the alphabet of MMM. The @ will be used to mark the position of a read/write head, and the $ will be used to delimit the parts of KKK's tape that represent the two tapes of MMM. For example, suppose that one of MMM's tapes contains the symbols "abb##cca" with the read/write head on the first b, and that the other tape contains "01#111#001" with the read/write head on the final 1. This configuration would be represented on KKK's tape as "$a@bb##cca$01#111#00@1$". To simulate one step of MMM's computation, KKK must scan its entire tape, looking for the @'s and noting the symbol to the right of each @. Based on this information, KKK can update its tape and its own state to reflect MMM's new configuration after one step of computation. Obviously, KKK will take more steps than MMM and it will operate much more slowly, but this argument makes it clear that one-tape Turing machines can do anything that can be done by two-tape machines.

We needn't stop there. We can imagine nnn-tape Turing machines, for n>2n>2n>2. We might allow a Turing machine to have multiple read/write heads that move independently on each tape. We could even allow two or three-dimensional tapes. None of this makes any difference as far as computational power goes, since each type of Turing machine can simulate any of the other types.1

Recursively Enumerable Languages#

We have used Turing machines to define Turing-acceptable languages and Turing-decidable languages. The definitions seem to depend very much on the peculiarities of Turing machines. But the same classes of languages can be defined in other ways. For example, we could use programs running on an idealized computer, with an unlimited amount of memory, to accept or decide languages. Or we could use nnn-tape Turing machines. The resulting classes of languages would be exactly the same as the Turing-acceptable and Turing-decidable languages.

We could look at other ways of specifying languages "computationally." One of the most natural is to imagine a Turing machine or computer program that runs forever and outputs an infinite list of strings over some alphabet Σ\SigmaΣ. In the case of Turing machines, it's convenient to think of a two-tape Turing machine that lists the strings on its second tape. The strings in the list form a language over Σ\SigmaΣ. A language that can be listed in this way is said to be recursively enumerable. Note that we make no assumption that the strings must be listed in any particular order, and we allow the same string to appear in the output any number of times. Clearly, a recursively enumerable language is "computable" in some sense. Perhaps we have found a new type of computable language. But no—it turns out that we have just found another way of describing the Turing-acceptable languages. The following theorem makes this fact official and adds one more way of describing the same class of languages:

Theorem: Let Σ\SigmaΣ be an alphabet and let LLL be a language over Σ\SigmaΣ. Then the following are equivalent:

  • There is a Turing machine that accepts LLL.
  • There is a two-tape Turing machine that runs forever, making a list of strings on its second tape, such that a string www is in the list if and only if w∈Lw\in Lw∈L.
  • There is a Turing-computable function f ⁣:{a}∗→Σ∗f\colon\{a\}^*\to\Sigma^*f:{a}∗→Σ∗ such that LLL is the range of the function fff.

While I will not give a complete, formal proof of this theorem, it's not too hard to see why it is true. Consider a language that satisfies the third property. We can use the fact that LLL is the range of a Turing-computable function, fff, to build a two-tape Turing machine that lists LLL. The machine will consider each of the strings ana^nan, for n∈Nn\in \Nn∈N, in turn, and it will compute f(an)f(a^n)f(an) for each nnn. Once the value of f(an)f(a^n)f(an) has been computed, it can be copied onto the machine's second tape, and the machine can move on to do the same with an+1a^{n+1}an+1. This machine writes all the elements of LLL (the range of fff) onto its second tape, so LLL satisfies the second property. Conversely, suppose that there is a two-tape Turing machine, MMM, that lists LLL. Define a function g ⁣:{a}∗→Σ∗g\colon\{a\}^*\to\Sigma^*g:{a}∗→Σ∗ such that for n∈Nn\in\Nn∈N, g(an)g(a^n)g(an) is the (n+1)th(n+1)^{th}(n+1)th item in the list produced by MMM. Then the range of ggg is LLL, and ggg is Turing-computable since it can be computed as follows: On input ana^nan, simulate the computation of MMM until it has produced n+1n+1n+1 strings, then halt, giving the (n+1)th(n+1)^{th}(n+1)th string as output. This shows that the second property implies the third property, so these properties are in fact equivalent.

We can also check that the second property is equivalent to the first. Suppose that LLL satisfies the second property. Consider a two-tape Turing machine, TTT, that lists the elements of LLL. We must build a Turing machine, MMM, which accepts LLL. We do this as follows: Given an input w∈Σ∗w\in\Sigma^*w∈Σ∗, MMM will simulate the computation of TTT. Every time the simulated TTT produces a string in the list, MMM compares that string to www. If they are the same, MMM halts. If w∈Lw\in Lw∈L, eventually it will be produced by TTT, so MMM will eventually halt. If w∉Lw\not\in Lw​∈L, then it will never turn up in the list produced by TTT, so MMM will never halt. Thus, MMM accepts the language LLL. This shows that the second property implies the first one.

The fact that the first property implies the second is somewhat harder to see. First, we note that it is possible for a Turing machine to generate every possible string in Σ∗\Sigma^*Σ∗, one-by-one, in some definite order (such as order of increasing length, with something like alphabetical order for strings of the same length). Now, suppose that LLL is Turing-acceptable and that MMM is a Turing machine that accepts LLL. We need a two-tape Turing machine, TTT that makes a list of all the elements of LLL. Unfortunately, the following idea does not work: Generate each of the elements in Σ∗\Sigma^*Σ∗ in turn, and see whether MMM accepts it. If so, then add it to the list on the second tape. It looks like we have a machine that lists all the elements of LLL. The problem is that the only way for TTT to "see whether MMM accepts" a string is to simulate the computation of MMM. Unfortunately, as soon as we try this for any string www that is not in LLL, the computation never ends! TTT will get stuck in the simulation and will never even move on to the next string. To avoid this problem, TTT must simulate multiple computations of MMM at the same time. TTT can keep track of these computations in different regions of its first tape (separated by $'s). Let the list of all strings in Σ∗\Sigma^*Σ∗ be x1x_1x1​, x2x_2x2​, x3x_3x3​, …. Then TTT should operate as follows:

  • 1: Set up the simulation of MMM on input x1x_1x1​, and simulate one step of the computation for x1x_1x1​

  • 2: Set up the simulation of MMM on input x2x_2x2​, and simulate one step of the computation for x1x_1x1​ and one step of the computation for x2x_2x2​.

  • 3: Set up the simulation of MMM on input x3x_3x3​, and simulate one step of each of the computations, for x1x_1x1​, x2x_2x2​, and x3x_3x3​.

  • …

  • n: Set up the simulation of MMM on input xnx_nxn​, and simulate one step of each of the computations, for x1x_1x1​, x2x_2x2​, \dots, xnx_nxn​.

and so on. Each time one of the computations halts, TTT should write the corresponding xix_ixi​ onto its second tape. Over the course of time, TTT simulates the computation of MMM for each input w∈Σ∗w\in\Sigma^*w∈Σ∗ for an arbitrary number of steps. If w∈Lw\in Lw∈L, the simulated computation for www will eventually end and it will appear on TTT's second tape. On the other hand, if w∉Lw\not\in Lw​∈L, then the simulated computation will never end, so www will not appear in the list. So we see that TTT does in fact make a list of all the elements, and only the elements of LLL. This completes an outline of the proof of the theorem.

General Grammars#

Next, we compare Turing machines to a completely different method of specifying languages: general grammars. Suppose G=(V,Σ,P,S)G=(V,\Sigma,P,S)G=(V,Σ,P,S) is a general grammar and that LLL is the language generated by GGG. Then there is a Turing machine, MMM, that accepts the same language, LLL. The alphabet for MMM will be V∪Σ∪{$,#}V\cup\Sigma\cup\{\$,\#\}V∪Σ∪{$,#}, where $ is a symbol that is not in V∪ΣV\cup\SigmaV∪Σ. (We also assume that # is not in V∪ΣV\cup\SigmaV∪Σ.) Suppose that MMM is started with input www, where w∈Σ∗w\in\Sigma^*w∈Σ∗. We have to design MMM so that it will halt if and only if w∈Lw\in Lw∈L. The idea is to have MMM find each string that can be derived from the start symbol SSS. The strings will be written to MMM's tape and separated by $'s. MMM can begin by writing the start symbol, SSS, on its tape, separated from www by a $. Then it repeats the following process indefinitely: For each string on the tape and for each production rule, x⟶yx\longrightarrow yx⟶y, of GGG, search the string for occurrences of xxx. When one is found, add a $ to the end of the tape and copy the string to the end of the tape, replacing the occurrence of xxx by yyy. The new string represents the results of applying the production rule x⟶yx\longrightarrow yx⟶y to the string. Each time MMM produces a new string, it compares that string to www. If they are equal, then MMM halts. If www is in fact in LLL, then eventually MMM will produce the string www and will halt. Conversely, if www is not in LLL, then MMM will go on producing strings forever without ever finding www, so MMM will never halt. This shows that, in fact, the language LLL is accepted by MMM.

Conversely, suppose that LLL is a language over an alphabet Σ\SigmaΣ, and that LLL is Turing-acceptable. Then it is possible to find a grammar GGG that generates LLL. To do this, it's convenient to use the fact that, as discussed above, there is a Turing-computable function f ⁣:{a}∗→Σf\colon \{a\}^*\to\Sigmaf:{a}∗→Σ such that LLL is the range of fff. Let M=(Q,Λ,q0,δ)M=(Q,\Lambda,q_0,\delta)M=(Q,Λ,q0​,δ) be a Turing machine that computes the function fff. We can build a grammar, GGG, that imitates the computations performed by MMM. The idea is that most of the production rules of GGG will imitate steps in the computation of MMM. Some additional rules are added to get things started, to clean up, and to otherwise bridge the conceptual gap between grammars and Turing machines.

The terminal symbols of GGG will be the symbols from the alphabet, Σ\SigmaΣ. For the non-terminal symbols, we use: the states of MMM, every member of Λ\LambdaΛ that is not in Σ\SigmaΣ, two special symbols <<< and >>>, and two additional symbols SSS and AAA. (We can assume that all these symbols are distinct.) SSS will be the start symbol of GGG. As for production rules, we begin with the following three rules:

S⟶<q0A>A⟶aAA⟶ε\begin{aligned} S&\longrightarrow <q_0A>\\ A&\longrightarrow aA\\ A&\longrightarrow\varepsilon \end{aligned}SAA​⟶<q0​A>⟶aA⟶ε​

These rules make it possible to produce any string of the form <q0an><q_0a^n><q0​an>. This is the only role that SSS and AAA play in the grammar. Once we've gotten rid of SSS and AAA, strings of the remaining terminal and non-terminal symbols represent configurations of the Turing machine MMM. The string will contain exactly one of the states of MMM (which is, remember, one of the non-terminal symbols of GGG). This tells us which state MMM is in. The position of the state-symbol tells us where MMM is positioned on the tape: the state-symbol is located in the string to the left of the symbol on which MMM is positioned. And the special symbols <<< and >>> just represent the beginning and the end of a portion of the tape of MMM. So, the initial string <q0an><q_0a^n><q0​an> represents a configuration in which MMM is in its start state, and is positioned on the first aaa in a string of nnn aaa's. This is the starting configuration of MMM when it is run with input ana^nan.

Now, we need some production rules that will allow the grammar to simulate the computations performed by MMM. For each state qiq_iqi​ and each symbol σ∈Λ\sigma\in\Lambdaσ∈Λ, we need a production rule that imitates the transition rule δ(qi,σ)=(τ,d,qj)\delta(q_i,\sigma) =(\tau,d,q_j)δ(qi​,σ)=(τ,d,qj​). If d=Rd=Rd=R, that is if the machine moves to the right, then all we need is the rule

qiσ⟶τqj\begin{aligned} q_i\sigma&\longrightarrow \tau q_j \end{aligned}qi​σ​⟶τqj​​

This represents that fact that MMM converts the σ\sigmaσ to a τ\tauτ, moves to the right, and changes to state qjq_jqj​. If d=Ld=Ld=L, that is if the machine moves to the left, then we will need several rules—one rule for each λ∈Λ\lambda\in\Lambdaλ∈Λ, namely

λqiσ⟶qjλτ\begin{aligned} \lambda q_i\sigma&\longrightarrow q_j\lambda\tau \end{aligned}λqi​σ​⟶qj​λτ​

This rule says that MMM changes the σ\sigmaσ to a τ\tauτ, moves left, and changes to state qjq_jqj​. The λ\lambdaλ doesn't affect the application of the rule, but is necessary to represent the fact that MMM moves left.

Each application of one of these rules represents one step in the computation of MMM. There is one remaining requirement for correctly simulating MMM. Since MMM's tape contains an infinite number of cells and we are only representing a finite portion of that tape, we need a way to add and remove #'s at the ends of the string. We can use the following four rules to do this:

<⟶<#<#⟶<>⟶#>#>⟶>\begin{aligned} <&\longrightarrow <\#\\ <\#&\longrightarrow <\\ >&\longrightarrow \#>\\ \#>&\longrightarrow > \end{aligned}<<#>#>​⟶<#⟶<⟶#>⟶>​

These rules allow blank symbols to appear at the ends of the string when they are needed to continue the computation, and to disappear from the ends of the string whenever we like.

Now, suppose that www is some element of LLL. Then w=f(an)w=f(a^n)w=f(an) for some n∈Nn\in\Nn∈N. We know that on input ana^nan, MMM halts with output www. If we translate the computation of MMM into the corresponding sequence of production rules in GGG, we see that for the grammar GGG, <q0an>⟹∗<hw><q_0a^n>\Longrightarrow^*<hw><q0​an>⟹∗<hw>, where hhh is the halt state of MMM. We want to show that S⟹∗wS\Longrightarrow^*wS⟹∗w. If we can just get rid of the <<<, the hhh, and the >>>, we will have that <hw>⟹∗w<hw>\Longrightarrow^*w<hw>⟹∗w and we can then deduce that S⟹∗wS\Longrightarrow^*wS⟹∗w for each w∈Lw\in Lw∈L, as desired. We can do this by adding just a few more rules to GGG. We want to let the hhh eliminate the <<<, move through the www, and then eliminate the >>> along with itself. We need the rules

<h⟶hh>⟶ε\begin{aligned} <h&\longrightarrow h\\ h>&\longrightarrow \varepsilon \end{aligned}<hh>​⟶h⟶ε​

and, for each σ∈Σ\sigma\in\Sigmaσ∈Σ,

hσ⟶σh\begin{aligned} h\sigma&\longrightarrow \sigma h \end{aligned}hσ​⟶σh​

We have constructed GGG so that it generates every string in LLL. It is not difficult to see that the strings in LLL are in fact the only strings that are generated by GGG. That is, LLL is precisely L(G)L(G)L(G).

We have now shown, somewhat informally, that a language LLL is Turing-acceptable if and only if there is a grammar GGG that generates LLL. Even though Turing machines and grammars are very different things, they are equivalent in terms of their ability to describe languages. We state this as a theorem:

Theorem: A language LLL is Turing acceptable (equivalently, recursively enumerable) if and only if there is a general grammar that generates LLL.

Turing Decidability#

In this section, we have been talking mostly about recursively enumerable languages (also known as the Turing-acceptable languages). What about the Turing-decidable languages? We already know that if a language LLL is Turing-decidable, then it is Turing-acceptable. The converse is not true (although we won't be able to prove this until the next section). However, suppose that LLL is a language over the alphabet Σ\SigmaΣ and that both LLL and its complement, L‾=Σ∗∖L\overline{L}=\Sigma^*\smallsetminus LL=Σ∗∖L, are Turing-acceptable. Then LLL is Turing-decidable.

For suppose that MMM is a Turing machine that accepts the language LLL and that M′M'M′ is a Turing machine that accepts L‾\overline{L}L. We must show that LLL is Turing-decidable. That is, we have to build a Turing machine TTT that decides LLL. For each w∈Σ∗w\in\Sigma^*w∈Σ∗, when TTT is run with input www, it should halt with output 1 if w∈Lw\in Lw∈L and with output 000 if w∉Lw\not\in Lw​∈L. To do this, TTT will simulate the computation of both MMM and M′M'M′ on input www. (It will simulate one step in the computation of MMM, then one step in the computation of M′M'M′, then one step of MMM, then one step of M′M'M′, and so on.) If and when the simulated computation of MMM halts, then TTT will halt with output 1; since MMM accepts LLL, this will happen if and only if w∈Lw\in Lw∈L. If and when the simulated computation of M′M'M′ halts, then TTT will halt with output 0; since MMM accepts LLL, this will happen if and only if w∉Lw\not\in Lw​∈L. So, for any w∈Σ∗w\in\Sigma^*w∈Σ∗, TTT halts with the desired output. This means that TTT does in fact decide the language LLL.

It is easy to prove the converse, and the proof is left as an exercise. So we see that a language is Turing-decidable if and only if both it and its complement are Turing-acceptable. Since Turing-acceptability can be defined using other forms of computation besides Turing machines, so can Turing-decidability. For example, a language is Turing-decidable if and only if both it and its complement can be generated by general grammars. We introduced the term "recursively enumerable" as a synonym for Turing-acceptable, to get away from the association with a particular form of computation. Similarly, we define the term "recursive" as a synonym for Turing-decidable. That is, a language LLL is said to be recursive if and only if it is Turing-decidable. We then have the theorem:

Theorem: Let Σ\SigmaΣ be an alphabet and let LLL be a language over Σ\SigmaΣ. Then LLL is recursive if and only if both LLL and its complement, Σ∗∖L\Sigma^*\smallsetminus LΣ∗∖L, are recursively enumerable.
